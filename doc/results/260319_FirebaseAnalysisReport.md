# Firebase 아키텍처 무결성 분석 및 리팩토링 전략 보고서

## 1. 개요
현재 프로젝트의 핵심 인프라인 Firebase 연동 로직의 안정성과 무결성을 분석한 결과입니다. 이전 네트워크 지연 이슈(`008-fix-save-hang-network`) 이후의 개선 사항이 잘 유지되고 있는지, 그리고 향후 확장성을 위한 리팩토링이 필요한지 검토합니다.

## 2. 분석 결과 요약

| 항목 | 상태 | 상세 내용 |
| :--- | :---: | :--- |

## 3. "수정 vs 재구축" 비용 분석

### 3.1. 전면 재구축 (Full Rewrite)
- **장점**: 모든 API에 일관된 미들웨어 패턴(Logging, Timeout, Error 정규화) 적용 가능.
- **단점**: 이미 검증된 `longPolling` 설정 및 `withTimeout` 로직 재구현 시 버그 발생 위험 높음. 약 3~5일의 작업 시간 소요 예상.
- **판단**: 현재 구조가 충분히 현대적(Firebase SDK v10+)이므로 비효율적임.

### 3.2. 정밀 수리 (Surgical Repair) - **채택**
- **장점**: 작동하는 기능을 유지하며 결함 지점(Timeout 누락, Kakao Auth)만 선택적으로 강화. 최소한의 리소스로 최대의 안정성 확보.
- **단점**: 공통 로직(Timeout 등)의 중복 호출이 발생할 수 있으나 유틸리티 함수로 관리 가능.
- **판단**: 현재 프로젝트 규모에서 가장 합리적이며 안전한 접근 방식임.

## 4. 향후 실행 로드맵 (Action Items)
1. **타임아웃 보완**: `interactionApi.ts` 및 `deleteCafe`에 `withTimeout` 적용.
2. **Kakao 로그인 해결**: Firebase Cloud Functions 또는 전용 백엔드를 통한 `signInWithCustomToken` 로직 구현 계획 수립.
3. **공통 에러 핸들러**: 모든 API 호출 결과를 정규화된 에러 객체로 변환하는 레이어 추가.

## 5. 결론
현재 Firebase 로직은 **"견고하지만 일부 마감이 덜 된 상태"**입니다. 전면 재구축보다는 발견된 결함 지점을 정밀 타격하여 수리하고, 미구현된 인증 브릿지 로직을 보완하는 방향으로 진행할 것을 권고합니다.
